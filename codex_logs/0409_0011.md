# 0409_0011

## 범위
- 다른 LLM이 제안한 `GT smoothing + weighted MSE + single-image overfit` 방향을 비판적으로 검토
- 현재 코드에 진단용 graph GT smoothing과 target 선택 경로를 추가
- `gt_sum=1`인 단일 양성 cluster에서 실제 overfit 실험 수행

## 비판적 검토와 수정 방향
- `GT smoothing` 자체는 타당하지만, 메인 파이프라인 정답을 곧바로 바꾸는 것은 부적절하다.
  - 이유: 픽셀 count 의미가 흐려지고, 최종 count 해석이 달라진다.
  - 조치: raw point GT는 유지하고, troubleshooting용 `y_edge_decay`를 별도 target으로 추가했다.
- `Weighted MSE`는 이번 문제에선 맞다.
  - 이유: 지금 해결하려는 문제는 count likelihood calibration이 아니라 `peak를 1 근처까지 세울 수 있느냐`이기 때문이다.
  - 따라서 single-graph overfit 진단에서는 `PoissonNLL`보다 `MSE`가 더 직접적이다.
- overfit protocol은 대표 cluster를 쓰면 안 된다.
  - 이유: 다중 선박 cluster에서는 smoothed target이 겹쳐 `target max > 1`이 될 수 있어 `pred_max > 0.9` 기준이 무의미해진다.
  - 조치: `gt_sum=1`인 양성 cluster를 자동 선택하는 `choose_overfit_cluster()`를 추가했다.
- overfit 진단에서는 정규화 성능이 아니라 fitting capacity를 봐야 하므로 `dropout=0`, `weight_decay=0`으로 고정했다.
- edge는 제안대로 undirected를 명시적으로 강제했다.

## 결론
- `최댓값이 0.3 근처에서 멈추는 현상`은 architecture가 절대 못 맞추는 문제가 아니라, 현재 `point GT + 약한 supervision` 조합의 문제로 보는 것이 맞다.
- 단일 양성 cluster overfit 기준에서는
  - raw point GT는 peak recovery에 실패했고
  - edge-smoothed GT만 써도 peak가 올라가며
  - edge-smoothed GT + weighted MSE(`positive_weight` 20 이상)는 `pred_max ≈ 0.997`까지 올라갔다.
- 따라서 troubleshooting 방향은 유효하다.

## 확정 방향
- 메인 파이프라인 raw GT는 유지
- troubleshooting / 후속 실험용으로만 `y_edge_decay` 경로를 유지
- overfit 문제 해결용 핵심 조합은
  - `undirected graph`
  - `edge-decay smoothed GT`
  - `weighted MSE`
  - `dropout=0`, `weight_decay=0`
  - `single-ship positive cluster overfit`
- 다음 단계는 이 조합을 patch set 수준으로 확장했을 때도 peak와 ranking이 같이 개선되는지 확인하는 것이다.
